# Фаза 4 — Масштабирование

**Цель:** перевести продукт из «один пользователь / один канал на инсталляцию» в
многопользовательскую, многоканальную и совместную платформу.

> Предусловие: Фазы 1–3 завершены (рабочий продукт с AI и Telegram).

---

## Направления

### Аутентификация и доступ
- [ ] **Refresh-токены** (сейчас только access-JWT; см. [Фаза 1](phase-1-core-api.md))
- [ ] Опциональный OAuth (Google / GitHub)
- [ ] Ротация и отзыв токенов (token blacklist)

### Мультиканальность
- [ ] Несколько каналов на одного пользователя
- [ ] Переключение активного канала в UI
- [ ] Изоляция данных по `channel_id` в дополнение к `user_id`

### Совместная работа
- [ ] Несколько пользователей на один канал (роли/права)
- [ ] Омниканальный чат — агрегация нескольких каналов

### Инфраструктура
- [x] **Очередь задач (Celery + Redis)** — **введена в [Фазе 3, шаг 4](phase-3-telegram.md#4b--отложенная-публикация-celery--redis)** (отложенный publish); далее импорт, AI, RAG-индексация можно подключить к той же `celery_app`
- [ ] Вебхуки для уведомлений
- [ ] Переход на Kubernetes при необходимости масштабирования
- [ ] Горизонтальное масштабирование бэкенда (stateless воркеры)

---

## Замечания по зависимостям

- **Очередь задач (Celery + Redis)** введена в [Фазе 3, шаг 4](phase-3-telegram.md)
  для планировщика публикаций (`app/celery_app.py`, сервис `celery-worker` в
  `docker-compose.yml`). Импорт, RAG и прочие фоновые задачи можно подключить к
  той же `celery_app` по мере готовности — Celery Beat пока не разворачивается
  (см. шаг 4b: реконсиляция при старте воркера вместо cron).
- **Мультиканальность** затрагивает модель данных (миграции) и фронтенд (выбор
  канала) — крупное изменение, планировать отдельно.
- **Совместная работа** требует пересмотра модели прав доступа (сейчас всё
  скоупится по `user_id`).

---

## Критерий завершения фазы

- Несколько пользователей и каналов работают изолированно и совместно.
- Тяжёлые операции вынесены в фоновую очередь.
- Инфраструктура готова к горизонтальному масштабированию.
