# Сборка контекста для AI-чатов

> Статус: **проектное решение (зафиксировано)**. Реализация — [Фаза 2](../backend/phases/phase-2-ai.md).

Документ описывает, **какие данные** попадают в запрос к LLM и **как** они собираются
для глобальных чатов и чатов поста. Сборка выполняется **на бэкенде** при каждом
AI-ответе; фронт передаёт текст сообщения и идентификаторы чата/поста.

---

## Области чата

Отдельного режима **home** нет — все диалоги с платформы считаются **глобальными**
(global chat). Дополнительно есть **чат поста** (post chat): тот же pipeline, но в
шаблон сводки добавляется текст поста.

| Область | Что входит в шаблон сводки |
|--------|----------------------------|
| Global chat | Профиль канала, все рубрики, аналитика канала |
| Post chat | То же + текст поста + аналитика поста |

---

## Слои контекста

При вызове LLM контекст складывается из независимых слоёв:

| Слой | Уровень API | Содержимое | Когда |
|------|-------------|------------|--------|
| **System prompt** | `system` | `systemPrompt` из AI-профиля | Каждый запрос |
| **Сводка (bundle)** | Скрытое `user` в primer | Шаблон: канал + рубрики + аналитика (+ пост) | Каждый запрос (с версионированием) |
| **Саммари диалога** | Внутри primer (тот же скрытый `user`) | Сжатие вытесненных пар сообщений | Когда история длиннее окна |
| **Окно диалога** | `user` / `assistant` | Последние видимые реплики | Каждый запрос |
| **RAG** | Дописывается к последнему `user` | Top-k заметок; веб-поиск | Каждый запрос (под флагом) |

**Не входит в сводку:**

- список заметок (только RAG);
- `systemPrompt` (только system-уровень);
- сырой полный текст длинной истории (заменяется саммари + окном).

---

## Три способа передачи контекста

### 1. Головная сводка (summary bundle)

- Собирается **шаблоном** (без LLM) из данных аккаунта на момент создания чата.
- Включает: поля вкладки **«Канал»** (tone, rules, phrases и т.д.), **все рубрики**
  канала, **аналитику канала и поста** (для post chat), **текст поста** (только post chat).
- При первом AI-запросе в чате в primer попадает **первое поколение** bundle
  (головная сводка).
- Вместе с bundle в primer добавляется **саммари диалога** (см. ниже), если оно уже
  есть; на старте чата саммари пустое.

### 2. Динамическая сводка

- При изменении канала или поста (fingerprint данных изменился) создаётся **новое
  поколение** bundle — пересбор **того же шаблона** с актуальными данными.
- **Саммари диалога в динамическую сводку не входит** — оно живёт отдельным блоком
  в primer (`SUMMARY_BUNDLE` и `CONTEXT_SUMMARY` — разные блоки в одном скрытом `user`).
- Поколения хранятся в метаданных чата (`rolling_summary_profile`, JSON). Отдельных
  полей «старая / актуальная» в сообщениях **нет**.

**Версионирование и «догоняние»:**

1. В primer для LLM всегда используется **stub-поколение** — сводка, которую модель
   «видит» первой в скрытом блоке.
2. При изменении данных появляется более новое поколение; оно **не подменяет stub
   мгновенно**, а «догоняет» в **ходе диалога**.
3. После **N user-turn** с момента якоря нового поколения (по умолчанию `N = 5`) новое
   поколение **заменяет** stub и становится первой сводкой в primer.
4. Пока догоняние не завершено, промежуточные поколения могут **вставляться** в окно
   диалога у соответствующих user-turn (floating generations).
5. **Двойное обновление** (два быстрых изменения канала/поста подряд): новые
   поколения **не подменяют stub сразу** при сохранении. В начале primer по-прежнему
   стоит старое поколение, пока новое не «созреет» через N user-turn после якоря.
   Когда G2 созревает — stub становится G2; когда G3 созреет — stub становится G3.
   Промежуточные поколения до созревания могут появляться в окне диалога (floating)
   у тех user-turn, где данные уже изменились.

Параметр `N` (число user-turn до созревания bundle) — конфигурируемый; по умолчанию **5**.

**Пример** (правки на 3-м и 4-м сообщении, `N = 5`):

| User-turn | Stub в начале primer |
|-----------|----------------------|
| 3–7 | G1 (канал до правок) |
| 8 | G2 (первое изменение созрело) |
| 9+ | G3 (второе изменение созрело) |

### 3. RAG (заметки и веб)

- **Заметки:** semantic retrieval по запросу пользователя; top-k фрагментов
  **дописываются** к тексту последнего `user`-сообщения перед вызовом LLM.
- **Веб:** отдельный web-search / Perplexity-подобный вызов; результат также
  попадает в контекст последнего `user`-сообщения (не в bundle).
- Включается флагом `RAG_ENABLED` и наличием пригодной модели (см.
  [runtime-modes.md](runtime-modes.md)).

## Саммари диалога (rolling summary)

Механика rolling summary:

- Назначение: сохранить смысл **вытесненных** из окна пар «пользователь → ассистент»,
  которые модель больше не видит в явном виде.
- Генерируется **LLM** (отдельный дешёвый вызов), инкрементально: первая пара →
  саммари; каждая новая вытесненная пара → обновление существующего саммари.
- Хранится в чате: `rolling_summary`, `rolling_summary_idx` (сколько сообщений уже
  учтено).
- В primer добавляется блоком `CONTEXT_SUMMARY` **внутри** скрытого `user` вместе с
  bundle, **только в головной части primer** (не пересобирается при смене канала).
- Формат: краткий текст от первого лица пользователя (до 6 предложений),
  с сохранением чисел и терминов.

Саммари появляется, когда длина валидной истории **превышает history window**
(по умолчанию **8** сообщений). Сообщения младше окна идут в LLM как обычный диалог.

## Скрытая двойка и видимый диалог

При создании чата из **композера** в БД сохраняется только **видимый** запрос
пользователя. Скрытая двойка **не пишется в историю чата** — собирается на лету
в `primer` перед каждым вызовом LLM:

```
[user]  СЛУЖЕБНЫЙ блок: SUMMARY_BUNDLE (+ CONTEXT_SUMMARY, если есть)
[assistant]  «Понял, учту» (или «Понял, учту при ответах.»)
… далее окно видимого диалога …
[user]  текущий запрос пользователя (+ RAG, если включён)
```

В UI пользователь видит **только** свои сообщения и ответы AI. Сводка и
подтверждение ассистента — служебные, в ленту не попадают.

Для уже существующего чата при новом сообщении primer строится так же; видимая
история не дублируется.

## Ветки диалога

История чата в TG Platform — **дерево**, не одна линия (см. `ChatMessage` в
[api-contracts.md](api-contracts.md), логика на фронте — `shared/lib/chatPaths.ts`).

### Структура

- У `user`-сообщения могут быть `userBranches[]`: каждая ветка — свой `text` и
  `continuation[]` (хвост диалога **ниже** этой точки).
- `activeUserBranch` — какая ветка **показана** в UI и считается «текущей линией».
- `ai`-сообщения с `variants` / `selectedVariant` (multi-mode) — это **не** ветки
  пользователя; в промпт попадает только выбранный вариант.

### Правило для LLM

**В контекст попадает только активная ветка** — линейный тред после
`flattenVisibleWithPaths(history)`.

| Механика | Неактивные ветки | Активная ветка |
|----------|------------------|----------------|
| Окно диалога (последние K сообщений) | нет | да |
| Счётчик user-turn (bundle, догоняние) | нет | да |
| Rolling summary (саммари) | нет | да |
| RAG к последнему `user` | нет | да |

Неактивные ветки хранятся в JSON чата для UI («что если спросить иначе»), но **не
смешиваются** в один промпт.

Сводка канала/поста (**bundle**) общая на весь чат — она от данных аккаунта, не от
ветки. Саммари диалога строится **только** по активной линии.

### Редактирование user-сообщения (форк)

Если у сообщения уже есть хвост (ответ AI и/или продолжение), правка текста **не
затирает** старую линию, а создаёт новую ветку:

```
Было:  U1 → A1 → U2 → A2

Редактируем U1 → U1'
       ├─ ветка 0: U1  → A1 → U2 → A2   (старая линия, сохранена)
       └─ ветка 1: U1' → A…            (новая, activeUserBranch = 1)
```

Для AI при активной ветке 1 в окно и саммари идут только `U1'` и хвост **ветки 1**.
Сообщения `U2`, `A2` остаются в ветке 0 и в промпт не попадают, пока пользователь
снова не переключится на ветку 0.

### Переключение ветки в UI

При смене `activeUserBranch` меняется видимый тред. Перед следующим вызовом LLM
бэкенд:

1. снова **линеаризует** активную ветку (`flattenVisibleWithPaths`);
2. пересчитывает user-turn, окно и floating bundle по этой линии;
3. пересобирает rolling summary **по активной линии** (без смешивания соседних веток).

При переключении ветки саммари не должно тянуть пары из «чужой» ветки — иначе модель
получит два несовместимых сценария в одном `CONTEXT_SUMMARY`.

### Реализация (Фаза 2)

На бэкенде перед сборкой primer:

```text
linear = flattenVisibleWithPaths(chat.history)
valid  = filter_alternating_roles(linear)
window = last K messages from valid
```

Вспомогательные поля веток (`userBranches`, `activeUserBranch`) в linearized
сообщениях для LLM **не передаются** — только `role` и текст (или выбранный variant
для `ai`).

## Сборка промпта перед вызовом LLM

Порядок:

```
1. system  ← systemPrompt из AI-профиля
2. primer  ← скрытая двойка (bundle + саммари → user; «Понял, учту» → assistant)
3. linear  ← flattenVisibleWithPaths(history) — только активная ветка
4. window  ← последние K сообщений из linear (по умолчанию K = 5)
5. inject  ← floating bundle-поколения у якорных user-turn (если догоняние идёт)
6. last user ← RAG (заметки / веб) дописывается к последнему user
7. → запрос к LLM (стриминг SSE)
```

**Фильтрация истории:**

- Начальные служебные `assistant` **до первого видимого** `user` в сырой истории
  отбрасываются (если когда-либо попадут в БД).
- Далее — проверка чередования ролей; битые записи пропускаются.
- В LLM уходит primer + усечённое окно, а не полный лог чата.

## Что передаёт клиент

Минимальный контракт (Фаза 2 уточнит поля):

- `chatId` — глобальный или постовый чат;
- `postId` — для post chat (чтобы бэкенд подтянул пост и аналитику поста);
- `text` — текст сообщения;
- `modelId` / активная модель — из композера (резолв ключа на бэкенде);
- флаги RAG / web-search — по настройкам профиля и env.

Клиент **не** собирает bundle, саммари и RAG — только отправляет сообщение;
бэкенд читает профиль, пост, заметки и метаданные чата из Postgres (+ overlay
для сид-аккаунтов на чтение).

## Параметры по умолчанию

| Параметр | Значение | Назначение |
|----------|----------|------------|
| `HISTORY_WINDOW` | 8 | Порог включения rolling summary |
| Окно в промпт | 5 | Сколько последних сообщений диалога в LLM |
| `SUMMARY_BUNDLE_CATCHUP_MESSAGES` | 5 | User-turn до «созревания» нового bundle |
| `SUMMARY_LIMIT` (саммари) | 6 предложений | Лимит rolling summary |

Числа можно изменить в конфиге TG Platform; логика остаётся той же.
