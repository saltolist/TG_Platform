# ADR-007: Режимы работы, резолв ключей AI и локальный overlay

## Статус
📝 Предложено (план; реализация — Фазы 1–2)

## Контекст

Один фронтенд (`frontend/`) обслуживает два контура сборки (GitHub Pages — витрина на
MSW; Docker — продукт на реальном Postgres) и должен поддерживать три пользовательских
режима:

1. **Презентация** — без входа, предзаполненные данные, общение с AI.
2. **Демо-аккаунт** — вход по демо-кредам, профиль/канал-имитация.
3. **Реальный аккаунт** — обычный пользователь со своими данными и ключами.

Возникли вопросы, требующие зафиксированного решения:
- Где живут данные демо/презентации в Docker и кто может их изменять?
- Откуда берётся реальный AI-ответ вместо заглушки и где хранятся ключи?
- Должен ли деплоер включать реальный AI без раздачи ключей демо-пользователям?
- Как поддержать RAG, не ломая режимы без бэкенда (Pages)?

## Рассмотренные варианты

**Данные демо/презентации в Docker**
- A. Полностью фронтовый seed (как Pages), бэкенд только для AI — расходится с целью
  «всё из Postgres».
- B. **(выбран)** Сид в реальные таблицы Postgres; правки демо/презентации — в
  локальный overlay, не в общую БД.
- C. Запись в общий демо-аккаунт — посетители портят данные друг другу.

**Хранение ключей AI**
- A. Только env (один ключ на инсталляцию) — нет BYOK для реальных аккаунтов.
- B. Только профиль (BYOK) — презентация без профиля не сможет звать реальный AI.
- C. **(выбран)** Профиль (реальные аккаунты) + env (инсталляция); демо хранит
  ссылку-переменную `env:<NAME>`, резолвимую в env-ключ.

**Стриминг**
- A. Возврат полного текста — проще, но без «живого» вывода.
- B. **(выбран)** SSE-стриминг; заглушка отдаётся тем же потоковым форматом.

## Решение

### Источник данных по контуру
- **Pages:** всё через MSW (без изменений).
- **Docker:** всё через Postgres; демо/презентация-данные предзаполняются в таблицы.

### Два предсозданных аккаунта (Docker)
- `presentation` — гость без профиля; доступ по `PRESENTATION_GUEST_TOKEN`.
- `demo-full` — вход `demo@mail.ru` / `Demo!2026`, профиль/канал-имитация.

### Локальный overlay
Для `presentation` и `demo-full`: чтения = сид из Postgres, записи = локальный слой
(localStorage), у каждого посетителя свой. Реальные аккаунты пишут напрямую в Postgres.

### Резолв ключей AI (на бэкенде)
1. Реальный ключ в профиле (BYOK — реальный аккаунт или введённый демо-юзером) → как есть.
2. Ссылка `env:<NAME>` (демо) → значение из env.
3. Пусто + презентация/демо → fallback на env-ключ по `provider`.
4. Ничего: презентация/демо → заглушка; реальный аккаунт → AI не работает (без заглушки).

Реальные ключи не уходят на фронтенд. Демо-пользователь видит замаскированную ссылку,
но **может ввести свой реальный ключ** поверх — тогда работает по нему (как BYOK).
Набор предзаполненных моделей демо-аккаунта определяется тем, ключи каких провайдеров
заданы в env; если ключей нет — предзаполнены заглушечные модели.

### Поведение по умолчанию
Свежий Docker-деплой без env-ключей → в презентации/демо AI-заглушки, у реального
аккаунта без ключа AI недоступен. Реальный AI включается, когда деплоер задаёт
env-ключи провайдеров (или пользователь вводит свой ключ).

### Изоляция реального аккаунта
Реальный аккаунт логикой не связан ни с презентацией, ни с демо: свои данные в
Postgres, свои ключи, без overlay и без заглушек.

### Стриминг
`POST /ai/reply/` → `text/event-stream`. Заглушка — тем же форматом. Только Docker;
на Pages поток имитируется на клиенте.

### RAG
Флаг `RAG_ENABLED` в env. Активен только при наличии пригодной реальной LLM-модели на
аккаунте (env-ключ для презентации/демо, BYOK для реального). Индексирует глобальные
заметки и заметки постов. Иначе retrieval пропускается.

## Последствия

- Демо/презентация в Docker «живые» (реальный Postgres), но не портятся: правки в
  overlay.
- Деплоер включает реальный AI/RAG одними env-ключами, не раздавая их демо-юзерам.
- `AssistantRepository` и чат-UI переходят на стриминговый контракт (SSE).
- Бэкенду нужны: гостевой токен → `presentation`, сидер аккаунтов, резолв ключей,
  LLM-клиент, опциональный RAG (pgvector).
- Контракт ключей: значения вида `env:<NAME>` — зарезервированный формат ссылки.
