# Режимы работы (презентация / демо / реальный аккаунт)

> Статус: **проектное решение (план)**. Описывает целевую механику. Текущий код
> реализует Фазу 1 частично; реальный AI/RAG — Фаза 2. См. [roadmap](../backend/roadmap.md).

Один и тот же фронтенд (`frontend/`) обслуживает три пользовательских режима. Режим
определяется **типом аккаунта**, а не только фактом входа. Источник данных зависит
от контура сборки (Pages vs Docker).

---

## Контуры сборки

| Контур | Данные | AI | RAG |
|--------|--------|----|-----|
| **GitHub Pages** (демо-витрина) | Всё через **MSW** в браузере | Только заглушки | Нет |
| **Docker** (продукт) | Всё через реальный **Postgres** | Заглушка или реальный LLM (по ключам) | По `RAG_ENABLED` + модель |

В Docker все демо-данные, которые сейчас живут в MSW, становятся **реальными строками
в таблицах Postgres** (предзаполняются при старте).

---

## Три режима (контур Docker)

### 1. Презентация — без входа

- Гость заходит без логина. Запросы к API несут гостевой токен
  (`PRESENTATION_GUEST_TOKEN`), бэкенд маппит его на предсозданный аккаунт
  `presentation`.
- **Нет профиля** (канал/настройки скрыты).
- Данные (посты, чаты, заметки) **читаются** из Postgres (`presentation`-аккаунт).
- **Правки гостя не пишутся в Postgres** — они хранятся в локальном overlay
  (localStorage), у каждого посетителя свои. См. [overlay](#локальный-overlay).
- AI: заглушка, **если** в env не заданы ключи провайдеров; при наличии env-ключей —
  реальный LLM (env-ключ трактуется как активная «реальная модель»).
- **Оркестратор не обязателен** для чата с ИИ — достаточно активной LLM-модели в композере
  (в отличие от демо и реальных аккаунтов).
- RAG: работает, если `RAG_ENABLED=1` и есть пригодная реальная модель (env-ключ);
  индексирует сид-заметки `presentation`-аккаунта.

### 2. Демо-аккаунт — вход по демо-кредам

- Вход: `demo@mail.ru` / `Demo!2026` → предсозданный аккаунт `demo-full` в Postgres.
- **Есть профиль и канал** (имитация реального аккаунта); модели можно переключать.
- Ключи в демо-данных — **замаскированные ссылки-переменные** (`env:<NAME>`), за
  которыми скрываются реальные ключи из env. Какие модели предзаполнены — зависит от
  того, ключи каких провайдеров заданы в env: заполняются именно эти модели. Если в
  env нет ни одного ключа — предзаполнены **заглушечные модели** (как в демо сейчас).
- Пользователь демо **может ввести свой реальный ключ** поверх переменной — и он
  будет работать (резолвится как обычный BYOK-ключ).
- Данные **читаются** из Postgres (`demo-full`); **правки — в локальный overlay**,
  не в общую БД (демо остаётся чистым для всех).
- AI: бэкенд резолвит ссылку-переменную → env-ключ (или реальный ключ, если введён).
  Есть пригодный ключ → реальный LLM; нет → заглушка.
- RAG: как в презентации — по `RAG_ENABLED` + наличие пригодной реальной модели;
  индексирует глобальные заметки и заметки постов аккаунта.

### 3. Реальный аккаунт — обычный пользователь

- Стандартный аккаунт, **логически не связанный** ни с презентацией, ни с демо.
- Регистрация/вход обычным образом, собственный аккаунт в Postgres.
- Профиль, ключи — **настоящие** (BYOK), вводятся пользователем и хранятся в БД.
- Данные **читаются и пишутся** в Postgres (никакого overlay).
- AI: по ключам из профиля. Есть валидный ключ → реальный LLM; **нет ключа — AI не
  работает** (заглушки нет; нет ключа — нет работы AI).
- RAG: по `RAG_ENABLED` + наличие реальной модели; индексирует глобальные заметки
  и заметки постов аккаунта.

## Резолв ключей AI

Бэкенд при запросе AI-ответа определяет ключ для активной модели по порядку:

1. **Реальный ключ в профиле** (BYOK — реальный аккаунт или введённый демо-юзером) →
   используется как есть.
2. **Ссылка-переменная** `env:<NAME>` (демо) → значение берётся из env бэкенда.
3. **Пусто + презентация/демо** → fallback на env-ключ провайдера по полю `provider`.
4. **Ничего пригодного** → для презентации/демо — заглушка; для реального аккаунта —
   AI не работает (заглушки нет).

Реальные env-ключи **никогда не уходят на фронтенд**: резолв и вызов LLM целиком на
бэкенде. Демо-пользователь видит только замаскированную ссылку (пока не введёт свою).

Набор предзаполненных моделей демо-аккаунта определяется тем, ключи каких провайдеров
заданы в env. Если в env нет ни одного ключа — предзаполнены заглушечные модели.

| Режим | Источник ключа | Поведение без ключа |
|-------|----------------|---------------------|
| Презентация | env (напрямую) | Заглушка |
| Демо | `env:<NAME>` → env (или свой BYOK) | Заглушка |
| Реальный | Профиль (BYOK) | AI не работает (без заглушки) |

## Локальный overlay

Для **презентации и демо** правки контента (создание/редактирование постов, заметок,
чатов) не должны попадать в общую БД. Решение:

- Чтения = данные из Postgres (сид аккаунта) **поверх которых** наложен локальный
  слой изменений.
- Записи = только в локальный overlay (localStorage), у каждого посетителя свой.
- При очистке localStorage пользователь возвращается к исходному сиду.

Для **реального аккаунта** overlay не используется — всё идёт напрямую в Postgres.

> Разделение по типу аккаунта: `presentation` и `demo-full` → overlay; реальные
> аккаунты → Postgres.

---

## Стриминг AI (SSE)

Реальные ответы LLM отдаются потоково через **Server-Sent Events**: `POST /ai/reply/`
возвращает `text/event-stream`. Когда пригодного ключа нет, в презентации/демо бэкенд
отдаёт заглушку тем же потоковым форматом (единый путь на фронте); у реального
аккаунта без ключа AI просто недоступен. Стриминг актуален только для Docker; на
Pages (MSW) поток имитируется на клиенте.

---

## RAG (опционально)

- Включается флагом `RAG_ENABLED` в env бэкенда.
- Активен только при наличии пригодной реальной LLM-модели на аккаунте
  (env-ключ для презентации/демо или BYOK для реального).
- Индексирует **глобальные заметки и заметки постов** аккаунта.
- При выключенном флаге или отсутствии модели — retrieval пропускается.
